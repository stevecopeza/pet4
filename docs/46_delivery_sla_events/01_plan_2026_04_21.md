## Overview

Delivery ticket SLA (`lifecycle_owner = 'project'`) is currently computed **inline on
every API call** in `TicketController::listTickets()`. This is correct and complete for
display purposes.

This sprint makes the SLA cron aware of delivery tickets so that it can:
- Persist the current SLA state to the DB (enabling the cron to detect transitions)
- Dispatch domain events when a delivery ticket first enters warning or breach
- Simplify `TicketController` by reading the stored value instead of recomputing

**Estimate: ~3–4h.**

## Current State (verified 2026-04-22)

### What works

`TicketController::listTickets()` (lines ~354–363) computes `sla_status` for delivery
tickets inline:

```php
} elseif ($ticket->lifecycleOwner() === 'project' && !$isTerminal) {
    $due = $ticket->resolutionDueAt();
    if ($due !== null) {
        if ($due < $now)                  $slaStatus = 'breached';
        elseif ($due < $warningThreshold) $slaStatus = 'warning';
    }
}
```

The UI badge is correct. This is the only path that sets delivery SLA status.

### What does not exist

- **No `sla_status` column** on `wp_pet_tickets` — status is never persisted.
- **`Ticket` entity** (`src/Domain/Delivery/Entity/Ticket.php`) has no `slaStatus()`
  getter or `$slaStatus` property. Constructor accepts SLA timing fields
  (`$slaId`, `$responseDueAt`, `$resolutionDueAt`, etc.) but no status field.
- **`SlaCheckService`** (`src/Application/Support/Service/SlaCheckService.php`) —
  the main loop skips project tickets entirely with a comment:
  ```php
  // Delivery (project) tickets: sla_status is computed inline in TicketController
  // from resolutionDueAt vs now — no clock state required.
  ```
  No `evaluateDeliverySla()` method exists.
- **No delivery SLA domain events** — `src/Domain/Delivery/Event/` contains only
  `MilestoneCompletedEvent.php`. Support equivalents (`TicketWarningEvent`,
  `TicketBreachedEvent`) exist under `src/Domain/Support/Event/` but are not reused.

## Design Decisions

### No separate clock state — the column is the store

For support tickets the SLA is a stateful clock (paused, resumed, reset) stored in
`wp_pet_sla_clock_state`. Delivery SLA is **stateless and deterministic** — it depends
only on `resolution_due_at` and `now`. Caching the computation in a column is the right
approach to enable transition detection without replicating the clock-state machinery.

**Decision: Option A — add `sla_status VARCHAR(20) NULL` to `wp_pet_tickets`.**

This collapses storage, transition detection, and controller simplification into one
migration. The column is authoritative; `TicketController` reads from it.

### Event semantics — fire once per transition

The cron runs every 5 minutes. Events must fire **once per transition**:
- `ok → warning` fires `DeliverySlaWarning` once
- `warning → breached` (or `ok → breached`) fires `DeliverySlaBreached` once
- Re-running the cron while the ticket is already in `warning` or `breached` → no event

This is achieved by comparing the new computed status to the stored `sla_status` before
dispatching, then updating the column.

### Where to put the new events

Parallel to the support events. Create under `src/Domain/Delivery/Event/`:
- `DeliverySlaWarningEvent.php`
- `DeliverySlaBreachedEvent.php`

### Step 5 — `SlaCheckService::evaluateDeliverySla()`

`src/Application/Support/Service/SlaCheckService.php`

Add new private method:

```php
private function evaluateDeliverySla(Ticket $ticket): void
{
    $due = $ticket->resolutionDueAt();
    if ($due === null) {
        return; // no deadline set — nothing to evaluate
    }

    $now              = new \DateTimeImmutable();
    $warningThreshold = $now->modify('+24 hours');

    if ($due < $now) {
        $newStatus = 'breached';
    } elseif ($due < $warningThreshold) {
        $newStatus = 'warning';
    } else {
        $newStatus = 'ok';
    }

    $previous = $ticket->slaStatus() ?? 'ok';

    if ($newStatus === $previous) {
        return; // no transition — no event
    }

    $ticket->updateSlaStatus($newStatus);
    $this->ticketRepository->save($ticket);

    if ($newStatus === 'warning') {
        $this->eventBus->dispatch(new DeliverySlaWarningEvent(
            ticketId: $ticket->id(),
            projectId: $ticket->projectId(),
            resolutionDueAt: $due->format(\DateTimeInterface::ATOM),
        ));
    } elseif ($newStatus === 'breached') {
        $this->eventBus->dispatch(new DeliverySlaBreachedEvent(
            ticketId: $ticket->id(),
            projectId: $ticket->projectId(),
            resolutionDueAt: $due->format(\DateTimeInterface::ATOM),
        ));
    }
}
```

Wire into the main loop — replace the delivery comment with:

```php
foreach ($activeTickets as $ticket) {
    if ($ticket->lifecycleOwner() === 'support') {
        $this->evaluate($ticket);
    } elseif ($ticket->lifecycleOwner() === 'project') {
        $this->evaluateDeliverySla($ticket);
    }
}
```

Check constructor — ensure `$this->ticketRepository` and `$this->eventBus` are
injected (they likely are already for support tickets; verify).

### Step 6 — Simplify `TicketController`

`src/UI/Rest/Controller/TicketController.php`

Replace the inline computation block (lines ~354–363):

**Before:**
```php
} elseif ($ticket->lifecycleOwner() === 'project' && !$isTerminal) {
    $due = $ticket->resolutionDueAt();
    if ($due !== null) {
        if ($due < $now)                  $slaStatus = 'breached';
        elseif ($due < $warningThreshold) $slaStatus = 'warning';
    }
}
```

**After:**
```php
} elseif ($ticket->lifecycleOwner() === 'project' && !$isTerminal) {
    $slaStatus = $ticket->slaStatus() ?? 'ok';
}
```

**Note:** The first cron run after the migration will populate `sla_status` for all
active delivery tickets. Until then, `slaStatus()` returns `null` and the controller
falls back to `'ok'`. This is acceptable — the UI currently shows no badge for `'ok'`
tickets, so there's no visual regression during the migration window.

## Warning Threshold

The `+24 hours` warning window used in `evaluateDeliverySla()` matches the existing
inline logic in `TicketController`. If this threshold becomes configurable in future,
both places must change. Consider extracting it to a named constant:

```php
private const DELIVERY_SLA_WARNING_HOURS = 24;
```
