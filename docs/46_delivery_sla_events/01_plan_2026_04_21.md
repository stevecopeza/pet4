## Overview

Delivery ticket SLA (`lifecycle_owner = 'project'`) is currently computed
**inline on every API call** in `TicketController::listTickets()`. This is correct
and complete for display purposes. The open question is whether the SLA cron
should also dispatch events when a delivery ticket breaches, enabling escalation
rules and advisory signals to react.

---

## Current State

### What works

`TicketController::listTickets()` (lines ~345–363) computes `sla_status` for
delivery tickets by comparing `resolution_due_at` to `now`:

```php
if ($ticket->lifecycleOwner() === 'project' && !$isTerminal) {
    $due = $ticket->resolutionDueAt();
    if ($due && $due < $now)                  $slaStatus = 'breached';
    elseif ($due && $due < $warningThreshold) $slaStatus = 'warning';
}
```

The result is included in every ticket API response. The UI shows the SLA badge
correctly for delivery tickets.

### What doesn't exist

- No `sla_status` column on `wp_pet_tickets` — status is never persisted.
- `SlaCheckService` (the 5-min cron) **skips project tickets** entirely
  (replaced the original `continue` with a clarifying comment; no
  `evaluateDeliverySla()` method exists).
- No cron events dispatched when a delivery ticket crosses a warning or breach
  threshold — so escalation rules and advisory signal generation cannot react to
  delivery SLA breaches automatically.

---

## Design

### Is a DB column needed?

For support tickets, the SLA state is a complex clock (start/stop/reset) stored in
`wp_pet_sla_clock_state`. For delivery tickets the state is **stateless and
deterministic** — it depends only on `resolution_due_at` and `now`. There is no
benefit to caching a value that is trivially recomputed on every read.

**Recommendation: no new column.** The inline computation in `TicketController`
stays and is authoritative. The cron's job is event dispatch only.

### Cron extension plan

Add `SlaCheckService::evaluateDeliverySla(Ticket $ticket): void`:

```
- Skip if ticket is terminal (done/closed/cancelled)
- Skip if resolution_due_at IS NULL
- Compute current status: 'breached' | 'warning' | 'ok'
- Compare to LAST KNOWN status (see tracking below)
- If status CHANGED since last check → dispatch SlaBreached / SlaWarning event
- Update the last-known status so the event fires only once per transition
```

### Last-known status tracking

Without a DB column there's no memory of what the cron last computed.
Two options:

**Option A** — store `sla_status` on the ticket row (simple, direct):
- Migration: `ALTER TABLE wp_pet_tickets ADD COLUMN sla_status VARCHAR(20) NULL`
- SlaCheckService reads the stored value, compares to current computed value,
  dispatches event on transition, writes new value.
- TicketController reads from the stored column instead of computing inline.

**Option B** — use a separate tracking table or transient (more complex):
- Not recommended.

**Recommendation: Option A** — add the column, it collapses three problems
(storage, comparison, controller simplification) into one migration.

### Events to dispatch

Reuse existing event infrastructure (`EventBus`). New domain events:

| Event | Payload | When |
|---|---|---|
| `DeliverySlaWarning` | ticket_id, project_id, resolution_due_at | First time status → 'warning' |
| `DeliverySlaBreached` | ticket_id, project_id, resolution_due_at | First time status → 'breached' |

These hook into the advisory signal generator and escalation rules if wired up.

## Priority Assessment

**This sprint is lower urgency than Sprint 45.** The system works correctly today —
delivery SLA is displayed accurately in the UI. The only thing missing is automatic
escalation/alerting when a delivery ticket breaches. If the product does not yet
have escalation rules wired to delivery tickets, this sprint can wait until that
integration is needed.

**Suggested sequencing**: do Sprint 45 (CRM deferred, 2h) first. Then assess
whether delivery SLA breach events are needed before implementing this.
