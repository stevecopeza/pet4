## Fix 1 — Add PET_ENV guard to `wp pet seed`

### Risk
`wp pet seed` has no environment check. Unlike `wp pet migrate` (which guards on `PET_ENV`),
the seed command can be run on any environment including production. Demo data writes directly
to live tables (`wp_pet_customers`, `wp_pet_tickets`, `wp_pet_leads`, etc.) with no prefix
separation. The only recovery path is `wp pet purge`, which requires the `seed_run_id` to be
retained.

The threat is not only human error. CI/CD pipelines with misconfigured environments, shared
hosting without `WP_ENV` set, or automated deploy scripts that run seeders are all realistic
scenarios.

### Fix
**File:** `pet.php`, seed command registration (lines ~221–242)

Mirror the guard that already exists for `pet migrate`:
```php
$env = \getenv('PET_ENV') ?: \getenv('WP_ENV') ?: '';
$allowed = ['local', 'development', 'dev'];
if (!\in_array($env, $allowed, true)) {
    \call_user_func('WP_CLI::error',
        'Seed command is restricted to local/dev environments. '
        . 'Set PET_ENV=local to proceed.'
    );
    return;
}
```

### Effort
~15 minutes. Zero risk.

## Fix 2 — Outbox Idempotency (unique constraint + transaction safety)

### Risk
Two independent failure modes in the event outbox:

**A) No unique constraint on `(event_id, destination)`**
The outbox table has only a non-unique `KEY event_id` index. Nothing prevents two rows for
the same event being enqueued to the same destination. If the EventBus dispatch path is
called twice (retry, concurrent request, duplicate event), both rows are picked up and the
external system receives the event twice.

**B) `markSent()` outside dispatch transaction**
The dispatch sequence:
1. `findDue()` + `claim()` — inside a transaction (sets `next_attempt_at` forward)
2. Transaction commits
3. External HTTP/API call executes
4. `markSent()` — **outside any transaction**

If the process crashes after step 3 but before step 4, the external call succeeded but
the row is never marked sent. When `next_attempt_at` expires, the row is eligible again
and the event is dispatched a second time.

**Impact:** Duplicate invoices in QuickBooks, duplicate notifications, duplicate financial
actions. In an authoritative operational system this is a trust failure.

### Fix A — Migration: unique constraint on outbox

New migration adding:
```sql
ALTER TABLE wp_pet_outbox
    ADD UNIQUE KEY uq_outbox_event_dest (event_id, destination);
```

**File:** `src/Infrastructure/Persistence/Migration/Definition/AddOutboxUniqueConstraint.php`

If duplicates already exist in the table, the migration deduplicates first:
```sql
DELETE o1 FROM wp_pet_outbox o1
INNER JOIN wp_pet_outbox o2
    ON o1.event_id = o2.event_id
    AND o1.destination = o2.destination
    AND o1.id > o2.id;
```

### Fix B — `markSent()` within the dispatch scope

**File:** `src/Application/Integration/Service/OutboxDispatcherService.php`

Option 1 (mark-then-call, at-least-once semantics): mark the row as sent *before* the
external call. If the call fails, retry logic handles it. The row will not be resent
unless a retry mechanism explicitly re-queues it.

Option 2 (wrap in try/finally): if at-most-once is needed, call `markSent()` inside a
try/finally that runs regardless of whether the external call throws.

The unique constraint (Fix A) is the backstop — even if `markSent()` fails, a duplicate
row cannot be inserted.

### Effort
~1.5–2 hours (migration + service change + test).

## Fix 3 — Time Entry Ownership Check in `correctEntry()`

### Risk
`POST /time-entries/{id}/correct` loads the original entry by `$id` with no ownership
check. `TimeEntry::createCorrection()` copies `$original->employeeId()` to the new entry.

Result: User A can create a correcting time entry attributed to User B's employee ID,
appearing in reports and billing exports as if User B made the change. This is audit
corruption and a financial integrity risk.

### Fix
**File:** `src/UI/Rest/Controller/TimeEntryController.php`, `correctEntry()` method

Before calling `createCorrection()`:
```php
$authenticatedEmployee = $this->employeeRepository->findByUserId(get_current_user_id());
if (!$authenticatedEmployee || $original->employeeId() !== $authenticatedEmployee->id()) {
    return new WP_REST_Response(['message' => 'Forbidden'], 403);
}
```

If the endpoint is admin-only (manage_options), the check can be relaxed to:
"admin may correct any entry" but the correction must record the admin's user ID as
the actor (an audit field), not silently inherit the original employee ID as creator.

### Effort
~1 hour.

## Fix 4 — AcceptQuoteHandler: Prevent Truth Drift ✅ Already implemented

**Finding during implementation (2026-04-22):** The entire `AcceptQuoteHandler::handle()`
method is already wrapped in `$this->transactionManager->transactional(...)` (lines 59–109).
Quote state write, event dispatch, and all ticket creation are inside one transaction. The
audit finding was based on incomplete code reading; the handler was already correct.

No change required. The original risk description is preserved below for the record.

### Risk (original assessment — does not apply)
`AcceptQuoteHandler` transitions the quote to ACCEPTED and then creates delivery tickets
as side effects. These two operations do not share a transaction boundary. If ticket
creation fails (or partially fails), the system's state diverges:

- Quote is permanently ACCEPTED
- No delivery tickets exist
- Reporting, SLA, and advisory layers that assume "accepted quote → delivery tickets exist"
  silently break

This is a **truth drift** failure: the system's representation of what happened has
diverged from the actual state of affairs.

### Fix
**File:** `src/Application/Commercial/Command/AcceptQuoteHandler.php`

Wrap the quote state write and all ticket-creation side effects in a single database
transaction:

```php
$this->wpdb->query('START TRANSACTION');
try {
    $quote->accept();
    $this->quoteRepository->save($quote);
    $this->provisionDeliveryTickets($quote);  // all ticket creation here
    $this->wpdb->query('COMMIT');
} catch (\Throwable $e) {
    $this->wpdb->query('ROLLBACK');
    throw $e;
}
```

Alternative approach: implement `ensureTicketsProvisioned(Quote $quote): void` as an
idempotent method that can be safely replayed if the initial run partially failed. This
is safer than a long transaction and supports recovery tooling.

### Effort
~2 hours (transaction wrapping + tests for the failure paths).

## Fix 5 — Quote Approval Enforcement

### Decision (2026-04-22)
Quotes must always go through an approval step before being sent to a customer. The
approver does not need to be a manager — the sales person who created the quote may
self-approve. But zero-approval dispatch is not permitted.

### Current State
The quote state machine permits `DRAFT → SENT` directly. The approval flow
(`DRAFT → PENDING_APPROVAL → APPROVED → SENT`) exists but is optional — any user can
call the send endpoint from DRAFT state and bypass approval entirely.

Additionally, `ApproveQuoteHandler` currently restricts approval to team managers and
escalation managers. A solo sales user with no manager cannot approve their own quote,
meaning if DRAFT→SENT is blocked, they have no path forward.

### Fix A — Block DRAFT→SENT in the state machine

**File:** `src/Domain/Commercial/ValueObject/QuoteState.php`

In `canTransitionTo()`, remove `SENT` from the valid transitions out of `DRAFT`.
Valid DRAFT transitions become:
```
DRAFT → submitForApproval → PENDING_APPROVAL
DRAFT → archive → ARCHIVED
```

`DRAFT → SENT` is no longer a valid transition. Calling `$quote->send()` from DRAFT
state throws a `DomainException`.

### Fix B — Allow self-approval by quote creator

**File:** `src/Application/Commercial/Command/ApproveQuoteHandler.php`

Extend the approver validation to accept the quote's owner/creator in addition to
managers:

```php
$isManager  = $this->isManager($approverId);
$isCreator  = ($quote->ownerId() === $approverId);

if (!$isManager && !$isCreator) {
    throw new \DomainException(
        'Approver must be the quote creator or a team manager.'
    );
}
```

The `Quote` entity needs to expose `ownerId()` if it does not already. `ownerId` is
the WP user ID of the staff member who created the quote.

### Fix C — UI: expose self-approval path

**File:** `src/UI/Admin/components/Quotes.tsx` (or wherever the quote action buttons live)

The "Submit for Approval" button should be visible to quote creators. After submitting,
the creator should see an "Approve" button (now permitted for them). This removes the
friction of the previous manager-only approval gate for small/solo teams.

If the portal quote builder also has an approval flow, the same button logic applies.

### Effort
~2–3 hours (entity state machine change + handler change + UI button logic + tests).
Tests are important here — state machine changes need positive and negative case coverage.
