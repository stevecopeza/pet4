## Item 1 — Lead: expose `opportunity_id` in domain + API

### Current State

- `wp_pet_leads.opportunity_id VARCHAR(36) NULL` — **exists in DB** (added by
  `CreateOpportunitiesTable` migration).
- `Lead` entity (`src/Domain/Commercial/Entity/Lead.php`) — **no `opportunityId`
  property**. The field is never read from the DB row.
- `SqlLeadRepository::hydrate()` — **does not extract** `opportunity_id`.
- `LeadController::serializeLead()` / enriched stdClass path — **does not include**
  `opportunity_id` in the API response.
- `Leads.tsx` admin UI — **no opportunity badge** on lead rows.

### What's Missing

Nothing structural — just plumbing the existing DB column through the layers:

1. **`Lead` entity**: add `private ?string $opportunityId` property, constructor param, getter.
2. **`SqlLeadRepository::hydrate()`**: set `opportunityId` from `$row->opportunity_id ?? null`.
3. **`LeadController`**: include `opportunityId` in both serialisation paths (enriched
   stdClass and entity).
4. **`Leads.tsx`**: add a small badge/pill on lead rows that have an `opportunityId`,
   linking to `#pipeline` for quick navigation.

### No migration needed

The column is already there.

## Item 2 — Lead status update when Opportunity created

### Current State

`CreateOpportunityHandler` creates the Opportunity with a `leadId` reference but does
not touch the Lead record. The Lead stays in its original status (`new` or `qualified`)
regardless of whether an Opportunity has been created from it.

### What's Missing

When an Opportunity is created from a Lead, the Lead's status should move to
`'qualified'` (if not already there or further along):

```php
// In CreateOpportunityHandler::handle(), after saving the Opportunity:
if ($command->leadId()) {
    $lead = $this->leadRepository->findById($command->leadId());
    if ($lead && $lead->status() === 'new') {
        $lead->qualify(); // transitions status to 'qualified'
        $this->leadRepository->save($lead);
    }
}
```

**Lead entity** needs a `qualify()` method (if one doesn't already exist) that sets
`$this->status = 'qualified'`.

**`CreateOpportunityHandler`** gets `LeadRepository` injected. DI binding already
exists for `LeadRepository`.

Also: the `Leads.tsx` "Create Opportunity" kebab action should additionally update the
opportunity_id on the lead row — this happens automatically once the handler does the
status update (the `opportunity_id` is already written to `wp_pet_leads` by
`CreateOpportunityHandler` via… wait — is it? Check: `CreateOpportunityHandler`
currently only creates the Opportunity record. It does NOT write back to
`wp_pet_leads.opportunity_id`.

**So the full back-link is also missing**: after saving the Opportunity, the handler
should also update `wp_pet_leads` SET `opportunity_id = $opportunity->id()` WHERE
`id = $command->leadId()`.

This is a single `$wpdb->update()` in the handler, or better: a `linkOpportunity()`
method on Lead entity + `LeadRepository::save()`.
