## Audit Summary

Two audit passes were conducted:

**Pass 1 — Conventional security audit**
Scope: auth callbacks, permission checks, WP-CLI gating, input escaping, capability model.
Finding: Auth exists on every REST endpoint. No unauthenticated surfaces. Capability checks
are present but coarse (manage_options for most admin endpoints).

**Pass 2 — Hostile integrity audit**
Scope: domain invariant bypass, event replay safety, cross-tenant data leakage, approval
bypass, read-side effects, documentation-to-code divergence.
Finding: Three structural risks (seed contamination, outbox idempotency, time entry
cross-user mutation). One latent architectural gap (data isolation). Core domain invariants
correctly implemented.

## Confirmed Vulnerabilities

### V-5: Time Entry Correction — No Actor Attribution for Admin Corrections
**Severity:** Medium (audit integrity risk)
**Status:** ⚠️ OPEN — accepted for current deployment; implementation deferred
**Detail:**
`correctEntry()` now blocks non-manager employees from correcting other users' entries (V-3 fix).
However, callers with `manage_options` and **no employee record** (pure admin accounts) are
explicitly allowed through — they bypass the ownership check. When an admin corrects a time
entry in this way, the resulting correction record retains the **original employee's ID** as
its `employee_id`. There is no `corrected_by_user_id` or `corrected_by_employee_id` stored.

In a billing export or an audit of time entries, the correction is indistinguishable from the
original employee having made it themselves. The actor is lost.

**Impact in current deployment:** LOW. There is one admin and one employee base; any correction
is attributable by elimination. If additional admin users are added, this becomes an audit gap.

**What a fix requires:**
- Add `corrected_by_user_id BIGINT UNSIGNED NULL` column to `wp_pet_time_entries` (migration)
- Update `TimeEntry::createCorrection()` to accept and store the actor WP user ID
- Update `TimeEntryController::correctEntry()` to pass `get_current_user_id()` into the constructor
- Update `SqlTimeEntryRepository` to persist and hydrate the new column
- Estimated effort: ~1.5–2h

**Revisit trigger:** before adding a second admin user, or before exposing time entry
correction to any auditing or billing reconciliation workflow.

### V-1 through V-4 (All resolved in Sprint 47)

### V-1: `wp pet seed` Has No Environment Guard
**Severity:** Critical (operational risk)
**Status:** ✅ FIXED — Sprint 47 (2026-04-22)
**Detail:**
Unlike `wp pet migrate` (which guards on `PET_ENV`), the seed command could be executed on
any environment. Data is written directly to live production tables. CI/CD misconfiguration
or shared hosting without `WP_ENV` set are realistic non-human-error trigger paths.
**Fix applied:** `pet.php` — added identical `PET_ENV`/`WP_ENV` guard to `wp pet seed`.
Command exits with error on any environment not in `[local, development, dev]`.

---

### V-2: Outbox — No Unique Constraint on `(event_id, destination)`
**Severity:** High (financial correctness risk)
**Status:** ✅ FIXED — Sprint 47 (2026-04-22)
**Detail:**
The event outbox table had no unique constraint on `(event_id, destination)`. The same
domain event could produce multiple outbox rows. Additionally, `markSent()` was called
outside the dispatch transaction — a crash between external call success and `markSent()`
left the row eligible for re-dispatch.
**Fix applied:**
- `AddOutboxUniqueConstraint` migration: deduplicates existing rows, then adds
  `UNIQUE KEY uq_outbox_event_dest (event_id, destination)`.
- `OutboxDispatcherService`: the four success-path writes (`recordInvoiceSnapshot`,
  `mappings.upsert`, `outbox.markSent`, `export->markSent`) are now wrapped in a single
  transaction, making post-dispatch bookkeeping atomic.

---

### V-3: Time Entry Cross-User Correction
**Severity:** High (financial integrity + audit risk)
**Status:** ✅ FIXED — Sprint 47 (2026-04-22)
**Detail:**
`correctEntry()` did not verify that the caller owned the original time entry. User A could
create a correcting entry attributed to User B's employee ID, appearing in billing exports
as if User B made the change.
**Fix applied:** `TimeEntryController::correctEntry()` now injects `EmployeeRepository` and
checks `$callerEmployee->id() !== $original->employeeId()` — returns 403 if the caller has
an employee record that doesn't match the entry's owner. Callers with `manage_options` but
no employee record (pure admin accounts) are not blocked.

---

### V-4: AcceptQuoteHandler Truth Drift
**Severity:** Medium (system consistency risk)
**Status:** ✅ CONFIRMED NOT PRESENT — Sprint 47 verification (2026-04-22)
**Detail:**
The audit finding was based on incomplete code reading. On implementation, `AcceptQuoteHandler::handle()`
was found to already wrap the entire operation — quote state write, event dispatch, and all
ticket creation — inside `$this->transactionManager->transactional()`. The truth drift risk
does not exist. No change was required.
**Finding:** AcceptQuoteHandler is correctly implemented. The audit was wrong on this item.

## Architectural Gaps (Require Design Decision)

### A-1: No Row-Level Data Scoping in Repositories
**Severity:** Architectural
**Status:** ACCEPTED — within current deployment scope. See multi-tenancy decision below.
**Detail:**
All major repository `findAll()` methods return data across all customers. No `WHERE
customer_id = ?` is applied automatically. IDOR exists on all detail endpoints. This is
not a single-file fix — it requires a scoping strategy applied to all repositories.
**Accepted because:** the confirmed deployment model is single-tenant, single MSP, with
no within-department customer splitting required. All admin users are trusted staff of
the same business. This is safe under that model.
**Revisit trigger:** adding a second customer-facing admin, or any portal user who should
be restricted from seeing other customers' data.

---

### A-2: Approval Enforcement — DECIDED AND FIXED
**Severity:** Medium
**Status:** ✅ FIXED — Sprint 47 (2026-04-22)
**Decision (2026-04-22):**
Quotes must always be approved before being sent to a customer. Approval does not
require a manager — the sales person who created the quote may self-approve. A quote
cannot be dispatched to a customer with zero approval steps.
**Fix applied:**
1. `QuoteState::canTransitionTo()` — `DRAFT → SENT` removed. Only valid path to SENT
   is now `DRAFT → PENDING_APPROVAL → APPROVED → SENT`.
2. `Quote` entity — `$createdByUserId` (WP user ID) property + getter added.
3. `AddCreatedByUserIdToQuotes` migration — `created_by_user_id BIGINT UNSIGNED NULL`
   added to `wp_pet_quotes`.
4. `CreateQuoteCommand` / `CreateQuoteHandler` — creator WP user ID flows through to entity.
5. `QuoteController::createQuote()` — records `get_current_user_id()` as creator.
6. `SqlQuoteRepository` — saves and hydrates `created_by_user_id`.
7. `ApproveQuoteHandler` — allows self-approval when `$quote->createdByUserId() === $approverUserId`.
   Legacy quotes (null creator) fall back to manager-only approval.
**UI deferred:** The "Approve" button in the quote UI is not yet shown to the creator —
  backend is ready; frontend update deferred to next UI sprint.

## Multi-Tenancy Decision Record

**Decision date: 2026-04-22. Decision: Single-tenant.**

### Confirmed model: Single MSP instance

PET is deployed as a single WordPress instance for a single MSP business. All admin
users are staff of that business. There is no requirement to isolate data between
independent MSP organisations.

**Consequence:** the current unscoped `findAll()` repositories are acceptable. All
admins are trusted to see all data. No `tenant_id` column or tenant injection layer
is required.

### Within-MSP staff scoping — DEFERRED, NOT REQUIRED NOW

The question was raised: should different staff members within the MSP (e.g. account
managers) be restricted to seeing only their assigned customers?

**Decision: Not required at this time.** All staff within the MSP can see all customers.
No departmental or role-based customer splitting is implemented.

**Note for future:** if a requirement emerges to split visibility by account manager
assignment, the correct implementation is:
- Repository methods accept an optional `$assignedToUserId` filter
- This is an additive change (no schema migration needed — `owner_id` / `assigned_to`
  columns already exist on some entities)
- Admin sees all by default; filter only applied when explicitly requested
- This is low-cost when the time comes

### Full multi-tenant (multiple MSPs in one instance) — NOT PLANNED

If PET were ever deployed as shared SaaS infrastructure serving multiple independent
MSP businesses, the cost would be significant:
- `tenant_id` column on all major tables (~15 migrations)
- Tenant context injected into every repository (not optional, not retrofittable easily)
- Separate authentication scopes per tenant
- This is an architectural rebuild, not a sprint

**This path is not planned.** If it ever becomes a requirement, it should be treated as
a major version migration, not an incremental change. Document that decision explicitly
at the time.

## Known Gaps in This Audit

Areas not fully investigated — may contain additional findings:

1. **Repository bypass enumeration** — Which handlers write directly via `$wpdb` bypassing
   the domain entity layer, and which invariants are skipped, was not fully traced. Any
   handler that bypasses `$entity->save()` is outside the domain's trust boundary.

2. **Portal endpoint scoping** — Whether portal-facing REST calls apply the same customer
   scoping as admin calls (or whether portal users can query any customer's data if they
   know the URL) was not verified.

3. **Activity feed `audience_scope` population** — Feed security depends on events being
   saved with correct `audience_scope` values. Whether all event publishers set this
   field accurately (vs. defaulting to `'global'`) was not audited.

4. **QuickBooks shadow table scoping** — Whether `wp_pet_qb_*` shadow tables are scoped
   per customer or aggregate was not verified.

5. **DRAFT→SENT approval decision** — RESOLVED (Sprint 47): DRAFT→SENT removed from state
   machine. Approval mandatory. Creator self-approval allowed.

6. **Time entry actor attribution on admin corrections** — IDENTIFIED as V-5 above.
   Accepted/deferred. Implementation path documented in V-5 entry.
