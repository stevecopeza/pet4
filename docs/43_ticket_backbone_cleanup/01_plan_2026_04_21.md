## Phase 4 — SLA Engine: Delivery Ticket Coverage

### Context
`SlaCheckService.php` (line 84) explicitly skips any ticket where `lifecycle_owner !== 'support'`:

```php
if ($ticket->lifecycleOwner() !== 'support') {
    continue; // SLA only for support tickets currently
}
```

Project delivery tickets (`lifecycle_owner = 'project'`) have their own workflow
(PLANNED → READY → IN_PROGRESS → BLOCKED → DONE → CLOSED) but receive no SLA treatment.

### Design

Delivery tickets need a different SLA model from support tickets:

| Dimension | Support | Delivery |
|---|---|---|
| Clock starts | Ticket created | Ticket moves to IN_PROGRESS |
| SLA measures | Response + resolution time | Delivery deadline (endDate / estimatedMinutes) |
| Breach condition | resolution_due_at exceeded | endDate exceeded while not DONE/CLOSED |
| Warning threshold | Configurable per tier | % of estimated budget consumed |

### Implementation Plan

**Step 4a — SLA Policy model for delivery**

Add a `delivery` SLA policy type in `SlaPolicy` / `SlaTier`. Delivery SLA does not use
`first_response_hours` — it uses a project-level `deadline` and per-ticket `estimated_minutes`.

**Step 4b — Extend `SlaCheckService`**

Replace the hard `continue` with a dispatcher:

```php
if ($ticket->lifecycleOwner() === 'support') {
    $this->evaluateSupportSla($ticket);
} elseif ($ticket->lifecycleOwner() === 'project') {
    $this->evaluateDeliverySla($ticket);
}
```

`evaluateDeliverySla()`:
- Skip if ticket status is DONE or CLOSED.
- Breach if `end_date IS NOT NULL AND end_date < now()`.
- Warning if `end_date IS NOT NULL AND end_date < now() + INTERVAL X DAYS`.
- Set `sla_status = 'breached' | 'warning' | 'ok'` on the ticket record.

**Step 4c — Surface in admin UI**

The existing SLA badge column in `SupportOperational.tsx` already reads `sla_status`. Once the
engine writes the value for project tickets, the badge will appear automatically. No UI changes
needed in Phase 4.

**Step 4d — Cron**

The existing `sla_automation` cron (every 5 min) calls `SlaCheckService`. No new cron needed —
just ensure the extended service covers both lifecycle owners.

### Estimated effort
Phase 4: 3–4 hours (service change + policy model + tests).

## Phase 4 Implementation Note

Phase 4 (SLA engine delivery coverage) was partially implemented:
- `SlaCheckService` was modified to **not hard-skip** project tickets (the `continue` was replaced with a comment)
